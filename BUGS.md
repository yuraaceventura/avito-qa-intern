1. 
Название: Неоднородность путей URL  
Описание: Пути URL для разных эндпоинтов имеют неоднородную структуру, которая не соответствует архитектуре REST.  
Фактический результат:  
    1. /api/1/item (создание объявления)  

    2. /api/1/item/:id (получение объявления)  

    3. /api/1/statistic/:id (статистика)  

    4. /api/1/:sellerId/item (объявления продавца)  
Ожидаемый результат:  
    1. /api/1/items (создание объявления)  

    2. /api/1/items/:id (получение объявления)  

    3. /api/1/statistics/:id (статистика)  

    4. /api/1/sellers/:sellerId/item (объявления продавца)  
Серьезность: Major (Архитектурная проблема, может привести к конфликтам роутинга)  
Приоритет: Low (Функционал API работает, баг не блокирует основные сценарии)  

2. 
Название: Отсутствие валидации отрицательных значений в числовых полях  
Описание: API позволяет создавать объявления с отрицательными значениями в полях price, likes, viewCount, contacts.  
Шаги:  
    1. Отправить POST запрос на /api/1/item с отрицательными значениями в полях price, likes, viewCount, contacts  
    2. Проверить ответ  
Ожидаемый результат:  
    1. Ответ возвращается со статусом 400 - Bad Request  
    2. В теле ответа содержится поле message, с сообщением о невалидном теле запроса  
Фактический результат:  
    1. Ответ возвращается со статусом 200 - ОК  
    2. Объявление создается  
Серьезность: Major (нарушена бизнес логика приложения)  
Приоритет: Medium (Может привести к проблемам, но основной функционал не затронут)  


3.   
Название: Несоответствие формата ответа в эндпоинтах /api/1/:sellerID/item и /api/1/item/:id  
Описание: В документации поле createdAt вынесено отдельно от объекта, но фактически в API оно находится внутри объекта.  
Ожидаемый результат:
```json
    {  
        "id": "<string>",  
        "sellerId": "<integer>",  
        "name": "<string>",  
        "price": "<integer>",  
        "statistics": {  
        "likes": "<integer>",  
        "viewCount": "<integer>",  
        "contacts": "<integer>"  
        },  
        "createdAt": "<string>"  
    }
```
Фактический результат:
```json
    {  
        "createdAt": "<string>",  
        "id": "<string>",  
        "name": "<string>",  
        "price": "<integer>",  
        "sellerId": "<integer>",  
        "statistics": {  
            "contacts": "<integer>",  
            "likes": "<integer>",  
            "viewCount": "<integer>"  
        }  
    }
```
Серьезность: Minor (Затронута только согласованность документации и API)  
Приоритет: Low (Функционал не нарушен, не критично для бизнеса)  


5.   
Название: Некоректный формат ответа при с ошибкой 400 в нескольких эндпоинтах  
Описание: По документации при ошибке 400 должен возвращатся ответ с бессмысленными данными Lorem Ipsum вместо информативного сообщения об ошибке. Затронутые эндпоинты:   
/api/1/item/:id  
/api/1/item  
/api/1/statistic/:id  
/api/1/:sellerId/item  

Ожидаемый результат:
```json
{  
  "result": {  
    "messages": {  
      "culpa_b92": "<string>",  
      "enim_24f": "<string>",  
      "mollit_aa": "<string>"  
    },  
    "message": "<string>"  
  },  
  "status": "<string>"  
}
```
Фактический результат:
```json
{  
    "result": {  
        "message": "<string>",  
        "messages": {}  
    },  
    "status": "<string>"  
}
```
Серьезность: Minor (Усложняет дебаггинг, но функционал не нарушен)  
Приоритет: Medium (Разработчики и пользователи, которые используют API, тратят много лишнего время на дебаггинг)  
  
5.   
Название: Неинформативное сообщение об ошибке в эндпоинте /api/1/item  
Описание: В теле ответа пишется "status": "не передано тело объявления" на любое неверное тело запроса. Появляется при вызове эндпоинта с полным, но не валидным телом  
Ожидаемый результат:
```json
{  
    "result" : {  
        "message" : "<текст ошибки>"  
    }  
    "status" : 400  
}  
```
Фактический результат:
```json
{  
    "result": {  
        "message": "",  
        "messages": {}  
    },  
    "status": "не передано тело объявления"  
}
```
Серьезность: Minor (Функционально все работает, проблема усложняет дебаггинг)  
Приоритет: Medium (Баг влечет за собой увеличениет времени разработки и отладки)
